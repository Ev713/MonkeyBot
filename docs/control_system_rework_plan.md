# MonkeyBot Control System Rework Plan

This plan replaces the current procedure-driven velocity controller with a
model-based control stack using the core ideas from Modern Robotics chapters 3,
5, 8, and 11: rigid-body transforms, twists, wrenches, Jacobians, constrained
dynamics, computed torque, task-space motion control, force control, and
impedance/admittance behavior.

The goal is not to make the code look academically formal for its own sake. The
goal is to remove the current "guess the motor speed" logic and replace it with
a control architecture where every command has a defined physical meaning:
desired pose/twist/acceleration, desired wrench, actuator torque/force, contact
constraint, or impedance.

## 1. Current System: What Must Be Replaced

### 1.1 Current control signal

The current control interface is:

```python
ControlSignal(
    extension: List[float],
    rotation: List[float],
    grip: List[int],
)
```

`extension[i]` is interpreted as a telescoping speed. `rotation[i]` is
interpreted as a desired angular rate. `grip[i]` is a discrete attach/release
request.

This is not torque control. It is closer to commanding idealized velocity
servos. The simulator then converts those commands to Pymunk `SimpleMotor`
rates and changing spring rest lengths.

### 1.2 Current actuator model

In `pymunk_simulator.py`:

- `RotationMotor` wraps `pymunk.SimpleMotor`.
- `RotationMotor.step()` blends measured angular rate and desired rate.
- `SpringMotor.step()` changes spring rest length directly.
- The controller never asks "what torque is required?"
- The controller never asks "what wrench do I need at the foot/body?"
- The controller never maps task-space errors through a Jacobian.

This means limb behavior depends on constraint stiffness, Pymunk solver
behavior, ad hoc damping, and procedure timing. It also explains the speed-up /
slow-down behavior you have been diagnosing: the controller is not regulating
torque, acceleration, or task-space error dynamics. It is repeatedly commanding
rates into a constrained physical system and hoping the constraints produce the
desired motion.

### 1.3 Current procedure model

The main control behaviors are in `procedure.py`:

- `AdjustAngle`
- `SmoothAdjustAngle`
- `AdjustLength`
- `Tracker`
- `MoveCenter`
- `DynamicCatch`
- `ReleaseAtPoint`

These procedures implement local heuristics:

- rotate toward a target point by sign of angle error;
- extend/retract by sign of length error;
- move center by updating target leg lengths;
- catch by greedily assigning feet to nearby points.

The procedure stack should not disappear completely. It should become a
high-level mode/trajectory state machine. It should stop directly commanding
motor rates. Low-level commands should be generated by model-based controllers.

### 1.4 Current jump model

`GeometricLauncher` predicts ballistic center trajectories and checks runway and
catch windows geometrically. The actual robot then tries to realize those
instructions using `MoveCenter`, `ReleaseAtPoint`, and `DynamicCatch`.

The missing link is a model-based controller that tracks the planned center
trajectory and foot trajectories using forces/torques. The current controller
does not enforce the predicted takeoff velocity. It only approximates it with
body-center motion procedures and release timing.

## 2. Book Concepts To Use

### 2.1 Chapter 3: Rigid-body motions, twists, and wrenches

Use chapter 3 as the foundation for representation:

- A body pose is a transform.
- A body velocity is a twist.
- A force/torque pair is a wrench.
- Twists and wrenches must be expressed in explicit frames.
- Adjoints transform twists between frames.
- Wrenches transform using the dual map so that power is invariant.

MonkeyBot is planar, so we can use planar versions:

```text
SE(2) pose:
    T = [ R(theta)  p ]
        [    0      1 ]

Planar twist:
    V = [omega, vx, vy]

Planar wrench:
    F = [tau, fx, fy]
```

Do not use raw `Vec2d.get_angle_between()` as the central representation of
robot orientation. Use explicit body transforms and relative transforms.

### 2.2 Chapter 5: Jacobians and wrench/torque duality

Use chapter 5 to define all mappings between actuator space and task space:

```text
task twist = J(q) qdot
joint force/torque = J(q)^T task_wrench
```

For MonkeyBot, the important Jacobians are:

- body pose Jacobian;
- foot endpoint Jacobian for each limb;
- stance/contact Jacobian for attached feet;
- center-of-mass or body-center Jacobian;
- optional relative foot-to-body Jacobian.

Every controller should answer:

1. What task-space error am I correcting?
2. What task-space wrench does that error imply?
3. What actuator torques/forces produce that wrench?

### 2.3 Chapter 8: Dynamics and constrained dynamics

Use chapter 8 to define the plant/control model:

```text
M(q) qdd + h(q, qdot) = tau + Jc(q)^T lambda
```

where:

- `q` is the generalized coordinate vector;
- `M(q)` is the mass matrix;
- `h(q, qdot)` includes Coriolis, centripetal, gravity, and friction terms;
- `tau` is actuator generalized force;
- `Jc(q)` is the contact constraint Jacobian;
- `lambda` is contact force/multiplier.

MonkeyBot has changing contact topology, so contact constraints are first-class
state, not side effects.

### 2.4 Chapter 11: Motion, force, hybrid, and impedance control

Use chapter 11 for controller design:

- joint-space computed torque for commanded joint trajectories;
- task-space computed torque for body/foot trajectories;
- force control for contact forces;
- hybrid motion-force control when some directions are constrained;
- impedance/admittance control for catching and contact-rich behavior.

The likely practical controller for this robot is not pure computed torque
everywhere. It should be a hybrid:

- stance/move center: task-space impedance with contact constraints;
- free limb movement: foot endpoint impedance through Jacobian transpose;
- jump runway: task-space body trajectory tracking with force limits;
- flight: ballistic prediction plus optional attitude control;
- catch: compliant foot endpoint impedance/admittance, then attach.

## 3. New Core Model

### 3.1 Generalized coordinates

Use an explicit generalized coordinate vector:

```text
q = [
    x_b, y_b, theta_b,
    alpha_0, l_0,
    alpha_1, l_1,
    alpha_2, l_2,
]
```

where:

- `(x_b, y_b, theta_b)` is the body pose in world coordinates;
- `alpha_i` is limb angle relative to body frame;
- `l_i` is limb extension;
- each foot endpoint position is derived from body pose and limb coordinates.

This is the conceptual model even if the Pymunk plant still stores body, leg,
and foot bodies separately. The controller should estimate this `q` from
Pymunk state each frame.

### 3.2 Planar forward kinematics

For foot `i`, define the body-frame foot vector:

```text
r_i_body = [l_i cos(alpha_i), l_i sin(alpha_i)]
```

World position:

```text
p_i_world = p_b + R(theta_b) r_i_body
```

This replaces scattered ad hoc vector logic in `Tracker`, `AdjustAngle`, and
`MoveCenter`.

### 3.3 Foot Jacobian

For each foot:

```text
p_i = p_b + R(theta_b) r_i(alpha_i, l_i)
```

The velocity is:

```text
pdot_i = J_i(q) qdot
```

`J_i` must include:

- derivative w.r.t. body translation;
- derivative w.r.t. body orientation;
- derivative w.r.t. limb angle;
- derivative w.r.t. limb extension.

This Jacobian is the central object for free-foot tracking and contact
constraints.

### 3.4 Body task Jacobian

For body pose tracking:

```text
V_body = [theta_dot, x_dot, y_dot]
```

The body task Jacobian is mostly selector rows from `qdot`, but it should still
be represented consistently as a Jacobian so that task wrenches can map to
generalized forces.

### 3.5 Contact constraint Jacobian

If foot `i` is attached to a grip:

```text
p_i(q) = p_grip
pdot_i(q, qdot) = 0
```

The velocity-level constraint is:

```text
J_contact_i(q) qdot = 0
```

For multiple attached feet:

```text
Jc(q) qdot = 0
```

This is how the controller knows which motions are allowed. Pymunk can still
enforce the constraints physically, but the controller should stop fighting them.

### 3.6 Wrenches and actuator forces

Define a generalized actuator vector:

```text
u = [
    tau_alpha_0, f_l_0,
    tau_alpha_1, f_l_1,
    tau_alpha_2, f_l_2,
]
```

where:

- `tau_alpha_i` is hip/limb rotation torque;
- `f_l_i` is axial leg extension force.

This replaces:

```text
rotation[i] = desired angular speed
extension[i] = desired extension speed
```

The Pymunk actuator layer should apply equal and opposite physical actions:

- rotation torque between body and leg;
- axial force between body and foot/leg along the current limb axis.

## 4. New Module Architecture

### 4.1 `monkey_bot/geometry.py`

Responsibilities:

- planar rotation matrices;
- SE(2) transform construction/inversion/composition;
- planar adjoint for twists;
- planar wrench transform;
- angle unwrap helpers;
- log/exp maps for planar pose error.

Initial functions:

```python
rot2(theta)
hat2(omega)
se2_from_xytheta(x, y, theta)
se2_inv(T)
se2_mul(A, B)
se2_log(T_error) -> PlanarTwist
adjoint_se2(T)
transform_twist(T_ab, V_b) -> V_a
transform_wrench(T_ab, F_a) -> F_b
```

### 4.2 `monkey_bot/model.py`

Responsibilities:

- define generalized coordinates;
- estimate `q`, `qdot` from simulator state;
- compute body/foot transforms;
- compute foot/body/contact Jacobians;
- compute approximate mass/gravity/friction model.

Core types:

```python
RobotState:
    q
    qdot
    body_pose
    body_twist
    feet_positions
    feet_velocities
    active_contacts

RobotModel:
    forward_kinematics(q)
    foot_position(q, limb_id)
    foot_jacobian(q, limb_id)
    body_jacobian(q)
    contact_jacobian(q, contacts)
    mass_matrix(q)
    bias_forces(q, qdot)  # gravity + Coriolis + damping/friction
```

Implementation note:

Do not try to derive the full exact mass matrix first. Start with a defensible
diagonal or block-diagonal approximation and validate it. The controller should
be written so `mass_matrix` and `bias_forces` can later be swapped for exact
dynamics.

### 4.3 `monkey_bot/actuators.py`

Responsibilities:

- convert generalized actuator force commands into Pymunk forces/torques;
- enforce torque/force saturation;
- optionally model motor bandwidth/friction;
- log actual actuator rates and control effort.

New signal:

```python
ActuatorCommand:
    limb_torques: list[float]
    limb_extension_forces: list[float]
    grip: list[int]
```

Optional later extension:

```python
ActuatorCommand:
    generalized_forces: np.ndarray
    contact_mode_requests: ...
```

Pymunk implementation:

- hip torque: apply `+tau` to leg body and `-tau` to robot body;
- extension force: apply equal/opposite forces along limb axis;
- grip attach/release remains discrete but is controlled by a contact manager.

Remove or bypass:

- direct `SimpleMotor.rate` as the primary actuator;
- direct spring rest-length stepping as the primary extension control.

Springs may remain as passive compliance, but not as the command path.

### 4.4 `monkey_bot/controllers/base.py`

Responsibilities:

- common control loop interface;
- saturation handling;
- stability diagnostics;
- conversion from task goals to actuator commands.

Interface:

```python
class Controller:
    def compute(self, state: RobotState, dt: float) -> ActuatorCommand:
        ...
```

### 4.5 `monkey_bot/controllers/joint_space.py`

Implement joint-level controllers:

1. PD torque controller:

```text
tau = Kp (q_d - q) + Kd (qdot_d - qdot)
```

2. Computed torque controller:

```text
tau = M(q) [qdd_d + Kd qdot_e + Kp q_e] + h(q, qdot)
```

3. Gravity compensated PD:

```text
tau = Kp q_e + Kd qdot_e + g(q)
```

Use these for controlled limb extension/rotation tests before using full
task-space control.

### 4.6 `monkey_bot/controllers/task_space.py`

Implement task-space controllers:

1. Foot endpoint impedance:

```text
F_i = Kp (p_d - p_i) + Kd (pdot_d - pdot_i) + M_task pdd_d
tau = J_i(q)^T F_i
```

2. Body pose/twist tracking:

```text
F_body = Kp X_e + Kd (V_d - V) + feedforward
tau = J_body(q)^T F_body
```

3. Multi-task stacking:

```text
tau = sum(J_task.T @ F_task for task in active_tasks)
```

Add priority/nullspace projection later. Do not start with full priority
control unless needed.

### 4.7 `monkey_bot/controllers/contact.py`

Responsibilities:

- track contact state;
- build contact Jacobian;
- classify stance/free/catch phases;
- project motion commands away from constrained directions;
- regulate contact forces when needed.

Modes:

```text
FREE_FOOT
STANCE_FOOT
CATCH_CANDIDATE
ATTACHED
RELEASING
```

Contact transition rules should be explicit. A foot is not just "near enough
therefore grip"; it should have:

- position tolerance;
- relative velocity tolerance;
- optional impact velocity threshold;
- optional normal/contact force threshold.

### 4.8 `monkey_bot/controllers/hybrid.py`

Implement hybrid motion-force control for stance:

- motion along allowed directions;
- contact force in constrained directions;
- avoid commanding impossible foot motion into attached contacts.

This is particularly important for `MoveCenter`. The current implementation
asks attached legs to change length/angle in ways that may fight fixed foot
constraints.

### 4.9 `monkey_bot/controllers/impedance.py`

Implement impedance behavior for catch and contact-rich movement:

```text
F = K (x_d - x) + B (xdot_d - xdot)
```

Use lower stiffness and higher damping during catch. Use higher stiffness during
stance. This should replace the current `DynamicCatch` behavior where trackers
chase targets with direct velocity commands.

### 4.10 `monkey_bot/trajectory.py`

Responsibilities:

- represent desired time-indexed trajectories;
- expose desired pose/twist/acceleration at time `t`;
- store phase boundaries and expected contact states.

Trajectory types:

- point-to-point minimum jerk;
- constant velocity;
- ballistic center trajectory;
- foot swing trajectory;
- stance body trajectory;
- jump runway trajectory;
- catch trajectory.

The geometric launcher should return a structured trajectory, not just
`start_point` and `takeoff_vector`.

## 5. New Simulator Interface

### 5.1 Replace `ControlSignal` at simulator boundary

Keep the old `ControlSignal` only temporarily behind an adapter. The simulator
should accept:

```python
ActuatorCommand(
    limb_torques=[...],
    limb_extension_forces=[...],
    grip=[...],
)
```

### 5.2 Add rich state feedback

Current `StateSignal` only contains:

```python
center_pos
feet_pos
active_grips
t
```

New state feedback should include:

```python
SimFeedback:
    body_pose
    body_twist
    body_velocity
    body_angular_velocity
    limb_angles
    limb_angle_rates
    limb_extensions
    limb_extension_rates
    feet_pos
    feet_vel
    active_grips
    contact_forces_estimate
    t
```

The controller should not finite-difference everything internally if Pymunk can
provide velocities directly.

### 5.3 Contact forces

Pymunk pivot constraints may expose impulse/force information through
constraint impulse APIs. If available, use this to estimate contact wrenches.
If not available, estimate from foot acceleration and actuator forces.

Contact force estimates are needed for:

- diagnosing failed catches;
- limiting grip loads;
- hybrid motion-force control;
- impedance tuning.

## 6. Control Behaviors After Rework

### 6.1 Free limb movement

Current behavior:

- release grip;
- set angular speed by sign of target angle;
- set extension speed by sign of length error.

New behavior:

1. Desired foot trajectory:

```text
p_d(t), pdot_d(t), pdd_d(t)
```

2. Compute measured foot state:

```text
p_i(q), pdot_i = J_i(q) qdot
```

3. Compute task-space force:

```text
F_i = Kp(p_d - p_i) + Kd(pdot_d - pdot_i) + M_task pdd_d
```

4. Map to generalized force:

```text
tau_q = J_i(q)^T F_i
```

5. Extract actuator torque/force components for limb `i`.

This should make limb motion smooth because desired acceleration and damping
are explicit. Speed oscillation becomes measurable as poor damping or actuator
saturation, not mysterious controller behavior.

### 6.2 Body center movement while feet are attached

Current behavior:

- compute next center point;
- update leg lengths/angles;
- attached foot constraints indirectly fight these commands.

New behavior:

1. Active contacts define constraints:

```text
Jc(q) qdot = 0
```

2. Desired body twist:

```text
V_body_d
```

3. Solve constrained least-squares for feasible generalized velocity or force:

```text
min ||J_body qdot - V_body_d||
subject to Jc qdot = 0
```

or in force form:

```text
tau = projected_task_wrench + contact_force_regulation
```

4. Use impedance to avoid fighting Pymunk constraints.

### 6.3 Attach / catch

Current behavior:

- `DynamicCatch` assigns target points greedily;
- foot trackers chase targets;
- grip attaches if within tolerance.

New behavior:

1. Predict reachable catch points using current body pose and foot Jacobians.
2. Select assignments by minimizing:

```text
distance cost
+ relative velocity cost
+ manipulability penalty
+ contact-load penalty
```

3. Track chosen foot endpoint with compliant impedance.
4. Attach only when:

```text
distance < epsilon_position
relative_speed < epsilon_velocity
impact angle acceptable
```

5. After attach, switch foot from free impedance to stance contact mode.

### 6.4 Jump runway

Current behavior:

- move body toward launch start;
- move body toward takeoff point;
- release at point;
- hope actual velocity matches geometric launcher.

New behavior:

1. Geometric launcher returns:

```text
center trajectory c_d(t)
center velocity cdot_d(t)
center acceleration cddot_d(t)
release time / release state
predicted ballistic trajectory
catch window
```

2. Stance controller tracks runway trajectory using attached support feet.
3. Controller explicitly tracks body velocity at release:

```text
||V_body_actual - V_body_takeoff|| < tolerance
```

4. Release is based on state, not only position:

```text
position close
velocity close
support forces within limits
```

### 6.5 Flight

During flight, no feet are attached.

Options:

- purely ballistic center dynamics;
- internal attitude control by moving limbs;
- keep limbs in catch-ready poses.

The initial version should avoid complex attitude control. Track free-limb catch
poses with low impedance/low force limits while allowing body ballistic motion.

### 6.6 Catch and post-catch stabilization

After one or more feet attach:

1. Switch attached feet to contact constraints.
2. Use hybrid/impedance control to avoid wrench spikes.
3. Continue free-foot catch tracking for unattached feet.
4. Once enough feet attach, regulate body pose to a stable target.
5. Declare success only after:

```text
required feet attached
body pose stable
contact forces below threshold
body velocity below threshold
```

## 7. Dynamics Model Strategy

### 7.1 Phase 1: Pymunk as plant, approximate model for control

Do not try to replace Pymunk immediately. Use Pymunk for forward dynamics and
contacts. Add a model-based controller on top.

The first mass model can be:

- diagonal body mass/inertia;
- diagonal limb angle inertias;
- diagonal extension inertias;
- gravity vector estimated from body/foot masses;
- damping/friction terms.

This is enough to implement stable PD/impedance control and will already be a
huge improvement over velocity commands.

### 7.2 Phase 2: Numerically identified model

Use simulation experiments to identify:

- effective inertia for each limb rotation;
- effective extension inertia;
- damping/friction;
- coupling terms between body and legs.

Fit parameters from logged actuator commands and observed accelerations.

### 7.3 Phase 3: Analytical dynamics

Derive a planar floating-base multi-branch dynamics model:

- body as floating base;
- each limb as a branch;
- foot constraints as holonomic constraints.

This is not a simple open chain. Modern Robotics open-chain formulas still
apply locally, but MonkeyBot is a floating-base tree with intermittent closed
chains. The constrained dynamics section is essential.

The analytical equation should converge toward:

```text
M(q) qdd + h(q, qdot) = B(q) u + Jc(q)^T lambda
Jc(q) qdd + Jcdot(q, qdot) qdot = 0
```

where `B(q)` maps actuator torques/forces into generalized forces.

## 8. Planner / Controller Boundary

The UPF planner should not care about torques, Jacobians, or wrenches.

Keep high-level actions:

- attach foot;
- release foot;
- transition link / jump;
- finish / stabilize.

But change action execution:

```text
Planner action -> Motion primitive -> Reference trajectory -> Controller -> ActuatorCommand
```

The old procedure classes should be replaced by motion primitive classes:

```python
class MotionPrimitive:
    def start(state): ...
    def reference(t, state): ...
    def contact_mode(t, state): ...
    def is_complete(state): ...
```

Examples:

- `MoveBodyPrimitive`
- `MoveFootPrimitive`
- `AttachFootPrimitive`
- `ReleaseFootPrimitive`
- `JumpPrimitive`
- `CatchPrimitive`
- `StabilizePrimitive`

## 9. Implementation Phases

### Phase 0: Freeze current behavior and metrics

Before changing controller code:

1. Keep the current experiment scripts.
2. Add baseline logs:
   - angle speed variance;
   - angle acceleration sign changes;
   - jump trajectory deviation by stage;
   - catch success/failure reason;
   - contact timing.
3. Save baseline CSVs in `data/baselines/`.

Exit criteria:

- We can reproduce current failures.
- We have quantitative metrics for "limb speed is inconsistent."

### Phase 1: Add geometry/model layer without changing behavior

Implement:

- `geometry.py`;
- `model.py`;
- state estimator from Pymunk state;
- foot/body Jacobians;
- finite-difference Jacobian tests.

Do not change control yet.

Exit criteria:

- `foot_position(q)` matches simulator feet.
- `foot_jacobian(q) @ qdot` matches measured foot velocity.
- Jacobian finite-difference tests pass.
- Wrench/velocity power identity holds:

```text
(J^T F)^T qdot == F^T (J qdot)
```

### Phase 2: Torque/force actuator backend

Implement `ActuatorCommand` and actuator application:

- hip torque;
- extension axial force;
- grip attach/release.

Add a compatibility adapter from old `ControlSignal` to new commands if needed.

Exit criteria:

- A single free limb can be accelerated and stopped with torque commands.
- Extension force produces expected extension acceleration.
- Equal/opposite forces conserve momentum when no grip is active.
- Torque/force saturation works.

### Phase 3: Joint-space torque control

Implement:

- limb angle PD torque control;
- limb extension PD force control;
- optional gravity compensation.

Replace `AdjustAngle`/`AdjustLength` internals with joint-space torque
controllers.

Exit criteria:

- Angle step tests are smooth.
- Speed variance and acceleration sign changes drop relative to baseline.
- No direct motor-rate commands remain in limb control.

### Phase 4: Foot endpoint impedance

Implement:

- foot task-space PD/impedance;
- foot Jacobian transpose mapping;
- free-foot swing trajectories.

Replace `Tracker` with endpoint impedance.

Exit criteria:

- Free foot reaches random target points smoothly.
- Path and speed are predictable.
- Catch approach velocity can be limited.

### Phase 5: Contact-aware stance controller

Implement:

- contact Jacobian;
- contact mode state;
- constrained body movement;
- stance impedance.

Replace `MoveCenter`.

Exit criteria:

- Body movement with attached feet no longer fights constraints.
- Contact feet remain near grips without large oscillatory actuator commands.
- Body pose tracking error is lower than baseline.

### Phase 6: Rework attach/catch

Implement:

- assignment cost for catch points;
- compliant catch controller;
- velocity-gated attach;
- post-attach stabilization.

Replace `DynamicCatch` and `Grabber` logic.

Exit criteria:

- Failed catches provide explicit reason:
  - no reachable assignment;
  - foot too fast;
  - missed target;
  - body unstable;
  - contact force too high.
- Catch success rate improves over baseline.

### Phase 7: Rework jump execution

Change `GeometricLauncher` output from:

```python
start_point, takeoff_vector
```

to:

```python
JumpTrajectory(
    runway_center_pose,
    runway_center_twist,
    takeoff_state,
    ballistic_prediction,
    catch_window,
    catch_targets,
)
```

Implement:

- runway trajectory tracking;
- takeoff velocity tracking;
- release based on state;
- flight/catch phase controller.

Exit criteria:

- Actual center trajectory deviation from predicted trajectory is measured and
  reduced.
- Jump timeout no longer hides whether catch is late or impossible.
- Jump success is evaluated after stabilization, not just first grip.

### Phase 8: Task-space computed torque / constrained dynamics

Once the approximate controllers are stable, add model-based feedforward:

- `M(q)`;
- `h(q, qdot)`;
- `Jc(q)`;
- constrained inverse dynamics;
- contact force estimates.

Exit criteria:

- Tracking improves with feedforward enabled.
- Feedforward can be disabled for A/B comparison.
- Controller remains stable under model error.

### Phase 9: Remove obsolete velocity-controller stack

Remove or quarantine:

- `RotationMotor` as velocity command path;
- `SpringMotor` as command path;
- `AdjustAngle`;
- `SmoothAdjustAngle`;
- `AdjustLength`;
- old `ControlSignal` where not needed;
- direct procedure-to-motor-rate logic.

Keep only if needed for legacy comparisons.

## 11. Expected Problems and Design Risks

### 11.1 Pymunk constraints may fight torque control

Pymunk constraints are solver-level objects. If controller torques are too stiff,
the solver and controller can fight each other.

Mitigation:

- use impedance, not infinite stiffness;
- add force/torque saturation;
- use contact-aware projections;
- tune gains from low to high.

### 11.2 The robot is a floating-base closed-chain system

Modern Robotics open-chain formulas are necessary but not sufficient. Gripped
feet create closed-chain constraints, and the body is not fixed to the world.

Mitigation:

- model contact constraints explicitly;
- use constrained dynamics from chapter 8;
- start with Pymunk-as-plant and approximate dynamics before exact analytical
  dynamics.

### 11.3 Contact transitions are discontinuous

Attach/release changes the system constraints instantly.

Mitigation:

- use mode transitions with guard conditions;
- ramp gains around contact events;
- require low relative velocity before attach;
- log impulses/contact loads.

### 11.4 Exact dynamics may be overkill initially

A full exact model is substantial work. A good impedance controller with a
reasonable approximate model may solve most observed issues first.

Mitigation:

- phase the work;
- validate each layer independently;
- keep feedforward optional.

### 11.5 Units and scaling must be cleaned up

Currently `cell_size` affects physics and visualization. This is dangerous for
experiments.

Mitigation:

- define world units explicitly;
- use `render_scale` only for viewing;
- keep controller gains in world/physical units;
- document all unit conversions.

## 13. First Concrete Implementation Milestone

The first milestone should be deliberately narrow:

1. Add `geometry.py`.
2. Add `model.py` with planar foot kinematics and Jacobians.
3. Add finite-difference Jacobian tests.
4. Add torque/force actuator backend.
5. Implement single-limb PD torque/force control.
6. Re-run the existing angle experiment.

Success is not "the whole robot climbs." Success is:

- no direct rate command needed for a limb angle step;
- actual angular speed is smooth;
- speed acceleration sign changes decrease sharply;
- overshoot and settling time are predictable from gains.

Only after that should the rework touch jump execution.

## 14. Recommended Controller Stack To Aim For

The final production architecture should look like:

```text
UPF plan action
    -> MotionPrimitive
        -> Reference trajectory + contact mode schedule
            -> StateEstimator
                -> RobotModel: kinematics, Jacobians, constraints
                    -> Controller:
                        - joint computed torque
                        - task-space impedance
                        - contact/hybrid force control
                    -> ActuatorCommand:
                        - hip torques
                        - extension forces
                        - grip requests
                    -> Pymunk plant
                        -> logged state, contacts, forces
```

The planner remains symbolic and discrete. The controller becomes physical and
continuous. The simulator becomes a plant receiving forces/torques, not an
idealized velocity actuator.
